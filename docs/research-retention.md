# 연구용 데이터 보관 설계 (탈퇴·동의 철회 시 가명처리 후 보관)

> 상태: 설계 결정 기록 (Phase 0, ERD 부속). **법률 자문 필요 항목 포함 — 여기는 설계 의도이며 법적 조언이 아니다.**
> 관련: `ERD`, `.claude/rules/security.md`(데이터 거버넌스 단일 출처), `.claude/rules/memory-state.md`(삭제 정책), `.claude/rules/cognitive-metrics*`(지표).

---

## 1. 결정

탈퇴·동의 철회 시 **완전 파기하지 않는다.** 대신 **신원과 연결을 복원 불가능하게 끊고(가명처리), 인지지표(수치)는 연구·통계 목적으로 보관**한다.

- 근거: 인지지표 시계열은 치매 조기징후 연구·서비스 개선에 가치가 큰 자산이다.
- 단, **재식별 위험이 있는 것(실명·연락처·발화 원문 등)은 파기**하고, **재식별 정보가 아닌 수치만 익명/가명 형태로 남긴다.**

---

## 2. 두-tier 신원 구조 (핵심)

신원(파기 대상)과 연구 주체(보관 대상)를 **처음부터 분리**한다. 그래야 철회 시 신원만 깨끗이 절단하고 수치를 남길 수 있다.

| 개념 | 내용 | 철회 시 |
|---|---|---|
| **elder_account** | 내부 PK + 실 PII(실명·연락처·로그인) | **파기** |
| **research_subject** | 연구용 가명 키(`subject_id`) — 인지지표를 이 키로 적재 | **보관**(신원 링크 끊긴 채) |
| **link (account ↔ subject)** | 둘을 잇는 매핑 | **파기**(= 재식별 절단) |

- **AI로 흘러가는 `target_user_id` = `subject_id`**(연구용 가명), elder_account PK가 아니다.
  → AI의 ChromaDB 임베딩·Kafka 인지지표가 처음부터 `subject_id`로 키잉되므로, 철회 시 매핑만 지우면 AI측 데이터도 자동으로 신원과 절단된다.
- 이것이 앞선 ERD 리뷰 #1(외부 가명 ≠ 내부 PK 분리)의 이유다. "연구 보관"을 택했으므로 이 분리는 **필수**다.

---

## 3. 철회 시 데이터 분류 (무엇을 파기/보관)

| 데이터 | 철회 시 | 이유 |
|---|---|---|
| 실명·연락처·계정(PII) | **파기** | 재식별 원천 |
| guardian·consent·account↔subject 매핑 | **파기** | 신원 링크 |
| 대화 원문 `turn`(stt_text/llm_response) | **파기** | 이름·개인사 포함 → 재식별 위험 |
| ChromaDB elder 임베딩 | **파기** | 원문 파생, 재식별 위험 |
| **cognitive_metric(수치 지표)** | **보관** (`subject_id` 유지, 신원 절단) | 비율·수치라 그 자체로 재식별 불가에 가까움 = 연구 자산 |
| session/turn 메타(시각·seq, **텍스트 제외**) | 보관 선택 | 시계열 구조가 연구에 필요하면 |

> rolling-window(M) 삭제는 이와 별개 — 운영 중 **ChromaDB 파생 캐시(오래된 임베딩)** 개수 정리다. RDBMS 원본 보관기간은 BE 정책 별도.

## 4. PIPA 관점 (설계 의도 — 법률 자문 필요)

- 보관 데이터는 **가명정보/익명정보**로서 **과학적 연구·통계** 목적으로 다룬다.
  - **가명정보 특례**(PIPA 제28조의2 등): 통계·과학적 연구·공익적 기록보존 목적이면 정보주체 동의 없이 처리 가능하나, **안전조치·재식별 금지(제28조의5)·결합 제한** 등 의무가 따른다.
  - **익명정보**(제58조의2): 재식별이 불가능하면 개인정보가 아니어서 PIPA 밖. 수치 지표만으로 개인을 특정할 수 없도록 하면 여기에 가깝다(가장 안전).
- **동의 고지 필수**: 최초 수집 동의(민감정보 별도 동의)에 **"가명처리 후 연구·통계 목적 보관·활용"**을 명시해야 근거가 선다.
- **민감정보(건강)**의 가명처리는 재식별 위험이 커 **엄격 관리·별도 보관·접근통제**가 요구된다.
- ⚠️ **[자문]** "가명 vs 완전 익명" 어느 수준까지 갈지, 시계열 보관이 재식별로 이어지지 않는지, 정보주체가 "파기"를 명시 요구할 때의 처리 — **전문가 자문 필요.**

---

## 5. ERD 반영 사항

1. `elder`(계정, PII) 와 `research_subject`(가명 키) **분리**. `elder.subject_id` FK.
2. **AI 노출 `target_user_id` = `subject_id`** (이미 만든 RAG elder write/read 스코프 키를 이 값으로).
3. `cognitive_metric`은 `subject_id`로 키잉(또는 subject 경유). `turn` 텍스트에 대한 FK는 **cascade 파기 대상**이되 metric은 남도록 설계.
4. 철회 처리 = elder_account·PII·turn텍스트·임베딩·매핑 **cascade 파기**, `research_subject`+`cognitive_metric` **보존**.
5. `consent`에 **연구 보관 동의 여부/버전** 포함.

## 6. 다운스트림 문서 갱신 필요 (doc 에이전트)

- `security.md`: 현재 "지체 없는 파기" 톤 → **"철회 시 신원 절단 후 가명 지표 연구 보관"**으로 보관·폐기 절 갱신. 가명정보 특례·재식별 금지·별도 보관 의무 명시.
- `memory-state.md`: 삭제 정책의 "이벤트 삭제" 를 **"PII·원문·임베딩 파기 + 지표는 subject_id로 가명 보존"**으로 정정. rolling-window는 캐시 정리로 유지.
- 동의 UI/문구(기획·법무): 연구 보관 조항 추가.

## 7. 미확정 / 자문

1. 가명 vs 완전 익명 수준 — 재식별 위험 평가 + 법률 자문.
2. 정보주체의 명시적 "파기" 요구 시 연구 보관 예외 가능 여부.
3. 연구 데이터 별도 저장소·접근통제 방식.
4. 시계열(장기) 보관이 재식별로 이어지지 않게 하는 기법(집계·노이즈 등).
